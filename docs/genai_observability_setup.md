# GenAI Observability for Splunk - Setup Guide

This guide explains how to install and use the **GenAI Observability for Splunk** app to visualize the token and latency metrics generated by this lab.

## 1. App Availability
**Good News!** In this lab environment, the **GenAI Observability for Splunk** app is pre-installed. It is automounted into the Splunk container via the `docker-compose.yml` file.

You do **not** need to download or install it manually unless you are starting from a completely fresh Splunk instance outside this project.

### If you need it manually anyway:
- **Splunkbase (Direct Link)**: [https://splunkbase.splunk.com/app/7382/](https://splunkbase.splunk.com/app/7382/)
- **GitHub (Source)**: [https://github.com/rootiq-ai/splunk_genai_observability](https://github.com/rootiq-ai/splunk_genai_observability)

## 2. Configure Data Ingestion
The lab is already configured to send data with the correct schema:
- **Index**: `main`
- **Sourcetype**: `genai:telemetry`
- **Fields**: Uses OpenTelemetry semantic conventions (`gen_ai.usage.input_tokens`, etc.)

## 3. Verify Dashboards
1.  Open the **GenAI Observability** app from the Splunk home screen.
2.  You should see active metrics for:
    - **Total Tokens** (Input/Output breakdown)
    - **Avg Latency** per request
    - **Model Performance** (LLama3 via Ollama)

## 4. Cost Configuration
Within the app, you can configure cost per 1k tokens for the "Ollama" provider to see real-time cost attribution for your digital twin simulations.

## Appendix: Comparison with Splunk AppDynamics

You may see documentation for **Splunk AppDynamics Observability for AI**. It is important to distinguish that product from the **GenAI Observability App** used in this lab.

| Feature | GenAI Observability (This Lab) | Splunk AppDynamics for AI | Splunk Observability Cloud |
| :--- | :--- | :--- | :--- |
| **Primary Use Case** | LLM/Agent Development & Labs | Enterprise Production APM | Real-time Cloud-Native Apps |
| **Key Metrics** | Tokens, Model Costs, Prompt Latency | Full-stack health, GPUs, Model Serving | Traces, Spans, Hallucinations, Bias |
| **Instrumentation** | Lightweight Python SDK (`genai-telemetry`) | AppDynamics Agent (Full-stack) | OpenTelemetry (OTel) Native |
| **Infrastructure** | Directly to Splunk HEC | Requires AppDynamics Controller | Cloud-native, SaaS-based |
| **Best For** | Prototyping, Local AI (Ollama), & Agents | Large-scale, mission-critical AI apps | Distributed microservices & RAG |

## When to use Splunk Observability Cloud?

While this lab uses Splunk Enterprise for local log-based analysis, **Splunk Observability Cloud** comes into play for production-scale, cloud-native AI applications.

### **Where?**
It lives in the cloud as a SaaS offering, designed for high-resolution streaming data from distributed services, LangChain workflows, and serverless LLM implementations.

### **Why?**
- **Real-time Resolution**: Identifies latency bottlenecks or "hallucinations" in seconds using streaming analytics. Splunk Observability Cloud is purpose-built for high-volume OTel spans where sub-second visibility is required.
- **Security & Risk**: Integrates with **Cisco AI Defense** to detect PII leakage or prompt injections in real-time. (While AI Defense *can* send logs to Splunk Enterprise, the Observability Cloud integration provides deeper context for agentic workflows).
- **AI Troubleshooting Agent**: Automatically correlates signals across traces and metrics to find the root cause of AI failures.

### **When?**
Move to Splunk Observability Cloud when:
1.  **Scaling to Production**: When your agent is no longer a "Hello World" and is handling thousands of requests per second. SOC's streaming architecture handles high-volume OTel data without the indexing latency found in traditional log platforms.
2.  **Using OpenTelemetry (OTel)**: If you've already started using OTel conventions (as we are in this lab), SOC is the native "home" for those traces, providing specialized visualizations for RAG and Agentic loops.
3.  **Security is Paramount**: When you need active defense against AI-specific threats (like prompt injection) at the observability layer.
